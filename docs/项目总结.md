## 一、项目目标

构建一个**教材向量化知识库**，当用户输入知识点时，系统自动从教材中检索相关内容，作为上下文传给大语言模型，生成高质量、结构化的7模块教学讲义。

### 核心流程

```
用户输入"一元一次方程"
    ↓
[向量检索] 从ChromaDB找到《七年级上册》第三章相关内容
    ↓
[Context组装] 提取定义、性质、例题、公式
    ↓
[LLM生成] 基于教材原文生成7模块讲义
    ↓
[渲染输出] HTML讲义，可编辑、可打印PDF
```

### 7模块讲义结构

1. **知识速览** - 核心概念、基本要求、应用目标
2. **重点精讲** - 定义、性质、推导、方法
3. **典例分析** - 例题含题目、分析、解答、答案、方法总结
4. **变式训练** - 变式题目及提示
5. **易错警示** - 常见错误、正确做法、原因分析
6. **巩固练习** - 基础题、提高题、拓展题
7. **知识总结** - 知识网络、核心要点、学习方法

## 四、根本问题分析

### 4.1 目录识别失败（核心问题）

**现状**：`extract_toc` 函数只匹配 `第X章` 或 `X.X` 格式

**无法识别的目录格式**：

| 教材类型 | 目录格式 | 当前正则 | 是否支持 |
|----------|----------|----------|----------|
| 数学（人教版） | `第二十六章` + `26.1` | `第X章` / `X.X` | ✅ 支持 |
| 历史（统编版） | `第一单元` + `第1课` | `第X章` / `X.X` | ❌ 不支持 |
| 语文（部编版） | `单元` + `课文` | `第X章` / `X.X` | ❌ 不支持 |
| 英语（外研版） | `Module` + `Unit` | `第X章` / `X.X` | ❌ 不支持 |
| 物理（人教版） | `第X章` + `X.X` | `第X章` / `X.X` | ✅ 支持 |

**结论**：只有约30%的教材（数学、物理）使用"章/节"格式，其余70%使用"单元/课"或其他格式。

### 4.2 封面/版权页过滤缺失

**现状**：从PDF第1页开始提取，混入大量封面信息

**混入内容示例**：
- 封面：书名、装饰图案、出版社标志
- 版权页：ISBN、定价、出版信息、印刷信息
- 编写人员名单：总主编、本册主编、编写人员、责任编辑
- 目录页：目录本身被当作正文，但目录结构未被提取

**影响**：
- 内容被垃圾信息污染（约30-50%的内容是封面/版权信息）
- 真正的教材正文被淹没
- 无法用于检索和向量化

### 4.3 页码切分错误

**现状**：按目录页码范围切分，但目录页码和实际PDF页码不一致

**问题**：
- 目录页显示"第1课 第2页"，但实际PDF第2页是封面
- 封面、前言、目录页占用实际页码，但教材正文从第10页开始
- 页码映射关系未建立，导致切分错误

### 4.4 内容截断不合理

**现状**：硬编码截断 `content[:5000]` 和 `content[:2000]`

**问题**：
- 截断可能导致内容不完整（如公式被截断）
- 没有按语义切分（如按段落、按小节）
- 不同章节长度差异大，统一截断不合理

### 4.5 解析逻辑过于简单

**当前流程**：
```
PDF文件
  → 提取前5页文本
  → 正则匹配 "第X章" 或 "X.X"
  → 按匹配到的页码切分
  → 保存JSON
```

**问题**：
- 没有处理没有目录页的PDF（ fallback 直接提取全文）
- 没有处理目录格式多样的情况
- 没有处理扫描版/文字版混合的PDF
- 没有处理分页符、页眉页脚等格式信息

## 六、修复方向

### 方案A：优化自动解析（推荐）

**目标**：使自动解析达到已有数据的质量标准

**具体措施**：

1. **支持多种目录格式**
   - 正则表达式扩展：支持 `单元`、`课`、`Module`、`Unit`、`Chapter` 等
   - 优先级策略：尝试多种格式，选择匹配度最高的
   - 学科特化：不同学科使用不同的目录识别策略

2. **过滤封面和版权页**
   - 识别封面特征：大标题、装饰图案、出版社标志
   - 跳过前N页（可配置，默认10页）
   - 过滤垃圾文本：ISBN、定价、编写人员等

3. **改进页码切分**
   - 建立目录页码和实际PDF页码的映射关系
   - 处理封面、前言、目录页占用的页码偏移
   - 使用相对页码而非绝对页码

4. **内容清洗和截断**
   - 按语义切分：按段落、按小节标题
   - 动态截断：根据内容长度调整截断点
   - 保留完整公式和图表描述

5. **增加后处理验证**
   - 检查章节数是否合理（至少2个章节）
   - 检查内容是否包含垃圾信息（关键词过滤）
   - 检查小节是否为空（空小节丢弃或标记）

### 方案B：接受低质量数据 + 后处理

**思路**：继续自动解析1361本，然后编写后处理脚本清洗

**后处理措施**：
- 删除包含 "ISBN"、"定价"、"编写人员" 等关键词的内容
- 合并只有一个"章节"的PDF（尝试重新切分）
- 人工补充关键教材的章节结构

**缺点**：
- 清洗工作量大，且不一定能完全修复
- 结构缺失（如小节为空）无法通过清洗弥补

### 方案C：混合策略

**思路**：不同学科采用不同策略

| 学科 | 策略 | 原因 |
|------|------|------|
| 数学、物理 | 自动解析（优化后） | 目录格式规范（章/节） |
| 历史、语文、英语 | 人工校准或半自动 | 目录格式多样，自动解析困难 |
| 其他 | 视情况 | 根据目录格式决定 |

## 七、关键决策

### 问题

1361本自动解析的教材目前全部是垃圾数据，继续运行只是浪费时间。

### 选择

1. **暂停解析，修复PDF解析逻辑**（推荐）
   - 优点：一次性修复，后续全量处理质量有保障
   - 缺点：需要额外开发时间（预计2-5天）
   - 风险：修复后可能仍不完美，需迭代

2. **继续运行，接受低质量数据，后续清洗**
   - 优点：不中断现有流程，可以"先跑完"
   - 缺点：清洗工作量大，可能无法完全修复
   - 风险：1361本垃圾数据占用大量磁盘空间，清洗困难

3. **人工处理，用已有数据的方式处理1361本**
   - 优点：质量最高，完全可控
   - 缺点：工作量巨大（1361本，每本需10-30分钟）
   - 风险：不可行，人力成本过高

### 建议

选择 **方案1（暂停并修复）**，原因：

- 自动解析是核心瓶颈，修复后全量处理才有价值
- 已有数据（rjb-9-2.json）证明"优质数据"的标准，自动解析必须达到这个标准
- 修复后可以继续从检查点恢复，不浪费已扫描的进度（Phase 1已完成，只修复Phase 2）
- 长远来看，修复后的解析逻辑可以复用，后续新增教材也能直接处理

### 下一步行动

1. **暂停当前正在运行的解析进程**（如果还在运行）
2. **修改 `preprocess.py` 的解析逻辑**：
   - 扩展 `extract_toc` 支持多种目录格式
   - 增加封面/版权页过滤
   - 改进页码切分
3. **用10-20本教材测试修复效果**（覆盖不同学科）
4. **验证通过后，从检查点恢复，继续全量解析**
